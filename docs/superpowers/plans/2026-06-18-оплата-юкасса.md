# Оплата ЮKassa — Implementation Plan

**Goal:** Подключить онлайн-оплату картой через ЮKassa поверх существующего контракта `PaymentGateway` (План 1). После оформления заказа со способом оплаты «картой онлайн» покупатель уходит на форму ЮKassa; webhook переводит заказ `pending_payment → paid`. Без уведомлений в CRM/Telegram (это План 5).

**Architecture:** Реализация `PaymentGateway` — класс `YooKassaGateway` с **инъекцией HTTP-клиента** (callable), чтобы юнит-тесты не ходили в сеть. Маппинг статуса платежа провайдера в статус заказа — отдельная **чистая функция** (тестируется без БД и сети). DB-операции оплаты — процедурные хелперы в `includes/order.php`, переходы статусов — через `can_transition()` (План 1), что даёт идемпотентность webhook'а. Подлинность уведомления — двухуровневая: проверка IP-источника (`verifySignature`) + повторный `getPayment` у провайдера (телу запроса не доверяем).

**Tech Stack:** PHP 8.3.31 (CLI: `C:\Users\USER\laragon\bin\php\php-8.3.31-nts-Win32-vs16-x64\php.exe`), PDO/MySQL 8.0, PHPUnit 9.6, cURL. API ЮKassa v3 (`https://api.yookassa.ru/v3`).

## Global Constraints

- Контракт `PaymentGateway` (План 1) не меняем; провайдер заменяем без правки заказов.
- Креды (`YOOKASSA_SHOP_ID`/`YOOKASSA_SECRET_KEY`) — только в `config.php` (gitignore). Тестовый магазин ЮKassa — для разработки.
- Суммы платежа берутся из заказа в БД (`orders.total`), не из клиента.
- Идемпотентность: создание платежа — `Idempotence-Key: order-<order_number>`; применение статуса — повторный вызов не ломает заказ.
- Webhook доверяет статусу из `getPayment`, а не из тела уведомления; 5xx-ответ → ЮKassa повторит.
- Тесты чистых/сетевых функций — PHPUnit, без реальной сети (HTTP инъектируется).

## Flow

1. Checkout (`payment_method=online`, шлюз настроен): `order_create` (статус `pending_payment`) → `createPayment` → `order_set_payment(payment_id)` → 303 на форму ЮKassa.
2. Покупатель оплачивает → ЮKassa редиректит на `return_url` = `/order_success.php?order=<n>`.
3. ЮKassa шлёт webhook на `/api/payment_callback.php` → проверка IP → `getPayment` → `order_apply_payment_status` → `pending_payment → paid`.
4. `order_success.php` показывает актуальный статус из БД.

Деградация: если шлюз не настроен или `createPayment` упал — заказ остаётся `pending_payment`, покупатель попадает на success, менеджер обрабатывает вручную.
